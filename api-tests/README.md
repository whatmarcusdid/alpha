|------|---------|

1. **Folder 1 upstream** — `GET /`, `POST /api/audit`, `POST /api/book-service/checkout` (retrieves real Stripe session metadata → `orderId`)
2. **Folder 2** — signed `POST /api/webhooks/stripe` using that metadata → verifies via `order-status`
3. **Folder 1 downstream** — `create-account` → **webhook re-fire** (seeds `fixSessions/{orderId}`) → `confirm-details` → `submit-access` → `order-status`
4. **Folder 3** — validation & edge cases (§3.5 negative paths)
5. **Folder 4** — performance / load spot checks (§4.4 — audit timing in Folder 1; burst stability here)
6. **Folder 5** — auth & account recovery (password reset, permission boundaries)
7. **Folder 6** — dashboard invite + #38 regression (Firestore state via smoke helper)
8. **Folder 7** — admin fix-job pipeline (progress → qa → rerun-checks → report → deliver → hosting-context)
9. **Folder 8** — site access re-request flow (admin request → grant/decline → revoke → cron expire)
10. **Folder 9** — analytics + notification validation (external delivery excluded — see below)
11. **Folder 10** — negative / security / abuse cross-section

## Folder 2 — Stripe Webhook Synthetic

Signs payloads with HMAC-SHA256 (`CryptoJS` in Newman sandbox — same algorithm as Stripe's `generateTestHeaderString` used in `e2e/support/stripe-webhook.ts`).

Metadata matches `buildSiteFixMetadata()` / checkout route — **sourced from the real Stripe Checkout session** after `POST checkout`:

1. Extract `cs_...` session ID from the returned checkout URL
2. `GET /v1/checkout/sessions/{id}` with `sk_test_` key (Postman `pm.sendRequest` in checkout test script — keeps single Newman run, works in Postman Desktop too)
3. Folder 2 webhook uses retrieved `metadata.orderId`, `auditLeadId`, `sku`, `normalizedEmail`, and real session `id`

**Trustworthiness:** This closes the checkout→webhook metadata handoff gap (#34/#35 class). The synthetic webhook validates the **same** `orderId` checkout wrote into Stripe, not a parallel UUID.

**fixSession seeding:** The first webhook runs before `create-account`, so `users/{uid}/fixSessions/{orderId}` does not exist yet (`triggerDashboardInvite` needs an existing user). Downstream re-fires the same event idempotently — matches Stripe retry behavior and triggers `ensureFixSessionOnDuplicateWebhook`.

## Folder 3 — Validation & edge cases

Runs after Folders 1–2. Assertions match **live route code** (not doc guesses):

| Request | Expected (from route) | Verified |
|---------|----------------------|----------|
| Checkout — missing `auditLeadId` | **400** `{ error }` (Zod) | ✓ |
| Checkout — invalid SKU | **400** `{ error }` (Zod enum) | ✓ |
| Create-account — existing email | **409** `"An account with this email already exists. Try signing in."` | ✓ |
| Confirm-details — missing `businessName` | **400** `{ error }` (Zod) | ✓ |
| Submit-access — incomplete payload | **400** `{ error }`; no password echoed | ✓ |
| Order-status — unknown UUID | **404** `{ success: false, error: "Order not found" }` | ✓ |

Existing-user test seeds a **second** unclaimed order via `POST /api/test/smoke/seed-pending-order` (direct Firestore write — **no** webhook/invite side effects) then retries `create-account` with the same `testEmail` from the happy path.

## Folder 4 — Performance / Load Spot Checks (§4.4)

Basic sanity only — **not** load testing. Runs after Folder 3 (needs `auditLeadId`, `testEmail` from Folder 1).

| Check | Where | What |
|-------|-------|------|
| Audit timing | **Folder 1** `POST /api/audit` | Hard fail if `responseTime ≥ 60s`; console **warn** (no fail) if `>15s` or `>20s` |
| Checkout burst | Folder 4 | 15× validation POST (missing `auditLeadId` → **400**, fast) — no 500+, no hang (>30s) |
| Checkout recovery | Folder 4 | Valid checkout **200** immediately after burst |
| Analytics burst | Folder 4 | 15× happy-path POST → all **200**, no hang (>15s) |
| Analytics recovery | Folder 4 | Valid analytics **200** immediately after burst |

**Different from Folder 10:** Folder 10's burst runner hunts **429** rate limits. Folder 4 checks **stability** (no persistent errors, clean recovery) — not a specific status code.

## Folder 7 — Admin Fix Jobs (full pipeline)

Runs after Folders 1–2 (seeds a real `fixSession` via customer funnel — not `POST /api/admin/fix-jobs`).

**Seeding choice:** Chained happy path produces real `orderId`/`fixJobId`, `userId`, `adminToken`, and `fixSessions/{orderId}` via webhook re-fire + submit-access — tests the actual customer→admin handoff.

**Route corrections vs doc wording:**

| Doc / assumption | Actual route |
|------------------|--------------|
| POST qa | **PATCH** `/api/admin/fix-jobs/:fixJobId/qa` |
| POST generate-report | **POST** `/api/admin/fix-jobs/:fixJobId/report/generate` |
| Stage transitions | **PATCH** `/api/admin/fix-jobs/:fixJobId/stage` (required between progress → qa → report_ready → delivered) |

**Sequence:** verify seed (`ready`) → hosting-context → stage `in_progress` → progress → unauthorized (401) → rerun-checks → stage `qa` → qa → stage `report_ready` → report/generate → deliver → hosting-context validation (400).

**Stage verification:** After each transition, `GET /api/admin/fix-jobs/:fixJobId?uid=` reads Firestore state server-side — asserts stages are not skipped.

**Prerequisite:** Storage emulator on `:9199` — `report/generate` uploads PDF via Admin SDK; without storage, generate returns 500 and deliver correctly returns 409.

**Local env finding (not an API bug):** `deliver` may return **500** `"Failed to send delivery email"` when Loops rejects PDF attachments on the current plan — PDF fetch + Firestore preconditions still pass; stage stays `report_ready` until delivery succeeds.

## Folder 5 — Auth & Account Recovery

Runs after Folder 3 (needs `testEmail`, `userToken`, `userId`, `orderId` from Folders 1–2).

**Password reset tokens:** App-generated 64-char hex docs in Firestore `passwordResets/{token}` — **not** Firebase Auth OOB codes. When `FIREBASE_AUTH_EMULATOR_HOST` is set **and** `NODE_ENV !== 'production'`, `request-password-reset` includes `devResetToken` in the response for test capture (production responses omit it).

**Order note:** Permission boundary tests run **before** password reset — Firebase revokes existing ID tokens after password change.

| Request | Expected (from route code) |
|---------|---------------------------|
| request-password-reset (known) | **200** `{ success: true, message: "If an account exists..." }` |
| request-password-reset (unknown) | **Same 200 + message** (anti-enumeration); no Firestore doc |
| GET reset-password?token= | **200** `{ valid: true, email }` or `{ valid: false, error }` |
| POST reset-password | **200** `{ success: true, message }` |
| POST reset-password (reused) | **400** `{ error: "...already been used..." }` |
| Protected route, no auth | **401** `{ error }` |
| Protected route, bad token | **401** `{ error }` |
| Admin route, non-admin token | **403** `{ error: "Forbidden" }` |

## Folder 8 — Site Access Re-Request Flow

Runs after Folder 7 (needs `userId`, `fixJobId`, `adminToken`, and a seeded `fixSessions/{fixJobId}`).

**Token for grant/decline:** `devAccessToken` is returned by `POST /api/admin/site-access/request` when `FIREBASE_AUTH_EMULATOR_HOST` is set **and** `NODE_ENV !== 'production'` (same dual gate as Folder 5 `devResetToken`). Production omits it — customer receives email links only.

**Firestore verification:** `GET/PATCH /api/test/smoke/site-access-request?requestId=` (emulator + localhost only) reads `siteAccessRequests/{id}` and can backdate `expiresAt` for cron testing.

| Request | Expected (from route code) |
|---------|---------------------------|
| POST admin/site-access/request | **200** `{ success: true, data: { requestId }, devAccessToken? }` |
| POST admin/site-access/request (duplicate pending) | **409** `"A pending access request already exists for this job"` |
| POST site-access/grant | **200** `{ success: true, data: { expiresAt } }` |
| POST site-access/grant (reused token) | **400** `"Invalid or expired access link"` (tokenUsed excludes doc from lookup) |
| POST site-access/decline | **200** `{ success: true }` |
| PATCH admin/site-access/:id/revoke | **200** `{ success: true }` (body: `{ uid }`) |
| GET cron/expire-access-requests | **200** `{ success: true, data: { expired } }` — auth: `Bearer ${CRON_SECRET}` when set; open when unset locally |

**Sequence:** request → duplicate 409 → grant → reuse 400 → new request → decline → new request → grant → revoke → new request → grant → backdate expiresAt → cron expire → verify `status=expired`.

## Folder 9 — Analytics / Notifications / External Side Effects

Runs after Folder 8 (needs `orderId`, `testEmail`, `userId` from upstream chain).

**Analytics (`POST /api/book-service/analytics`):** Always returns **200** `{ success: true }` — invalid payloads are silently dropped (no 400). Tests document this as a **FINDING** rather than asserting doc-specified validation errors. Happy path verified via emulator smoke helper `GET /api/test/smoke/analytics-event?marker=`.

**Notification routes — validation-only in automated suite** (stops before Slack/Loops/Notion):

| Route | Included test | Why safe |
|-------|---------------|----------|
| `POST /api/notifications/new-user` | Missing fields → **400** | Validates before `SLACK_WEBHOOK_URL` lookup |
| `POST /api/notifications/dashboard-ready` | Missing email → **400** | No Loops call |
| `POST /api/notifications/account-created` | Missing email → **400** | No Slack/Notion call |
| `GET /api/notifications/new-user` | **405** | Method guard |

**Explicitly excluded** (would post to real Slack / send Loops / write Notion):

- `POST /api/notifications/new-user` with valid body → **real Slack**
- `POST /api/notifications/dashboard-ready` with valid email → **real Loops email**
- `POST /api/notifications/account-created` with valid email → **real Slack + Notion update**
- `GET|POST /api/test/weekly-digest` → Stripe + Notion fetch + **real Slack**
- `GET /api/cron/weekly-sales-digest` → same as weekly digest cron

**Diagnostic smokes included:**

- `GET /api/hello` — plain-text health
- `GET /api/test-sentry-error` — intentional **500**; **posts to real Sentry** (delivery needs manual dashboard check)

## Folder 10 — Negative / Security / Abuse Cases

Runs last (needs `userId`, `orderId`, `fixJobId`, `userToken`, `adminToken`, `testEmail` from upstream). **Revokes `userToken` at end of folder** via password reset — do not chain anything after Folder 10 without re-auth.

| Case | Route sample | Expected (from route code) |
|------|--------------|----------------------------|
| Malformed JSON | `POST /api/audit`, `POST /api/book-service/checkout` | **400**, not 500 |
| Oversized field | `POST /api/audit` (max 128 on businessName) | **400** |
| Oversized field | `PATCH confirm-details` (no max in schema) | **FINDING** if not 400 |
| Missing auth | `PATCH confirm-details` | **401** |
| Invalid/revoked token | admin GET, confirm-details after reset | **401** |
| Wrong HTTP method | `GET /api/book-service/checkout`, `GET notifications/new-user` | **405** |
| Duplicate submit | `POST create-account` on claimed `orderId` | **409** |
| Rate-limit burst | 8× `POST create-account` `{}` | **429** if Upstash reachable; else **FINDING** fail-open |
| Unknown ID | `GET order-status`, `GET admin/fix-jobs` | **404** / **400** |

**Every test** asserts response bodies contain no stack traces, `node_modules`, or `/Users/` paths.
